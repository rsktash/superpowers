The Architecture of Attention: Re-Engineering Superpowers into a High-Level Workflow Skills System for LLM Behavioral GovernanceThe progression of large language model (LLM) integration into autonomous software engineering workflows has encountered a persistent failure mode known as attention drift. This phenomenon represents a fundamental architectural limitation where the model’s focus on primary objectives and constraints decays as the context window populates with intermediate tool outputs, environment logs, and conversational history. While current frameworks like the superpowers skills system utilize discipline anchors such as Red Flags and Iron Laws to prevent catastrophic failures, these mechanisms are primarily reactive or negative in nature. To achieve high-quality, reliable output in long-horizon tasks, the system must undergo an architectural shift toward task-as-directive governance. This paradigm transforms the "beads" work items from mere documentation into self-prompting instruments designed to anchor the LLM’s generative attention on the specific working task through Context Anchors, machine-verifiable Acceptance Gates, and topological Attention Maps.Theoretical Foundations of Positional Bias and Context RotThe internal mechanics of transformer-based LLMs are inherently biased by the position of information within the input sequence. Research into the "lost in the middle" phenomenon demonstrates a distinctive U-shaped performance curve, where models exhibit the highest accuracy when relevant information is placed at the absolute beginning (primacy bias) or absolute end (recency bias) of the context window. As an agentic session progresses, the initial system prompt and high-level project goals are inevitably pushed toward the center of the context, where the model’s effective recall can degrade by more than 30%.This degradation is exacerbated by the quadratic scaling of self-attention mechanisms, which creates billions of pairwise token relationships that dilute the signal of the original directive. The writing-plans skill must move beyond human-readable checklists and toward machine-executable directives that define the exact boundaries of the LLM’s focus.Architecting Tasks as Directive InstrumentsThe fundamental insight of a high-level workflow system is that when an LLM creates tasks for its future self, it is not writing a status report; it is performing prompt engineering. High-quality output is directly proportional to how well the task body controls the generative attention of the executor. A task-as-directive contains four critical anchoring components: Context Anchors, Acceptance Gates, Negative Constraints, and Drift Detectors.The Role of Context AnchorsA Context Anchor is more than a persona or a summary; it is a "why" statement that explains the task's role in the larger plan. In complex engineering tasks, LLMs often lose the thread of why a specific change is being made, leading to "tunnel vision" on mechanics while violating the root purpose of the feature. By restating the parent goal and the task's specific role in the plan (e.g., "This task implements the validation logic required for the API security layer"), the system keeps the agent’s reasoning aligned with the high-level intent.Acceptance Gates and Machine-Verifiable CriteriaThe "Acceptance Gate" replaces subjective completion signals with objective, machine-checkable criteria. An agent should never report a task as "done" based on its own internal "vibe" or a successful-looking code generation. Instead, the writing-plans skill should define specific, observable signals: files that must exist, exports that must be present, and tests that must pass.This shift toward Time-To-Acceptance (TTA) as the primary productivity metric acknowledges that the cost of AI development is not in drafting (which is near-zero), but in verification and rework. When the writing-plans skill decomposes a feature, it must generate a prose narrative of how each task relates to the others. This involves modifications to the writing-plans, executing-plans, and subagent-driven-development components.Upgrading the writing-plans SkillThe writing-plans skill should be enhanced with a "self-review" step focused on attention quality. Before finalizing a plan, the agent must verify that each task could be executed with zero additional context beyond the task body and the files it references. This ensures that the task is truly self-contained and avoids the "context poisoning" that occurs when an agent must search through a long history of design discussions to understand a current requirement.The Execution Loop and Attention RefreshThe current execution loop (bd ready → read task → execute → bd close) is insufficient for long tasks. The executing-plans skill must implement an "Attention Refresh" mechanism between steps. After every 3-4 tool calls or after every significant step in the task body, the agent should re-read the Acceptance Gate. This re-injection of the completion criteria into the recency bias of the model keeps the generative attention weighted toward the actual goal rather than the immediate output of the last command.Subagent Isolation and Context EngineeringHigh-quality output requires that each task be executed in a clean environment. The subagent-driven-development skill should be the default for executing implementation plans. By spawning a fresh subagent for each task, the system prevents "history pollution"—the accumulation of failed attempts, old error logs, and discarded ideas that can distract the model.Effective context engineering also dictates how information is presented to these subagents. Following the principles of KV-cache optimization, the system should push all dynamic information (like the current task state and recent logs) to the end of the prompt while keeping static instructions and tool definitions at the beginning. This stable prefix ensures that the model’s core reasoning "persona" is preserved while its active attention is focused on the immediate task.Advanced Verification and the Pre-AGI Verification TaxThe shift toward autonomous agents introduces a "verification tax." In traditional workflows, human drafting is slow but generally reliable; in AI-agent workflows, drafting is instant but reliability is uneven. By re-engineering the superpowers skills to include directive task formats, topological attention maps, and frequent attention refreshes, we move from a system that "hopes" the LLM follows instructions to a system that "governs" its behavior at runtime.This governance is enforced through the interplay of high-level specs (the PRD), externalized memory (Beads), and rigorous verification (Acceptance Gates). The result is a high-level workflow system that dictates attention, rejects drift, and ensures that every token generated by the LLM is anchored to the specific task at hand. This architecture provides the technical rigor required to scale AI-driven development from simple scripts to complex, secure, and maintainable software systems.
